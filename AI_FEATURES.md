# 커리어 저금통 AI 기술 지도

> 결선 발표의 `관측 데이터 → FactSheet → UserProfile → 서비스 개인화` 흐름을 실제 구현 기준으로 정리한다. 코드가 최종 진실의 원천이며, 구현과 로드맵을 구분해 표기한다. 기준일은 2026-07-20이다.

## 한 문장 정의

커리어 저금통의 AI는 금융 금액을 임의로 생성하는 모델이 아니라, 불규칙한 소득과 행동을 근거와 함께 판독해 선택지를 좁히는 로컬 판단 계층이다.

> 판단은 AI · 원화는 산수 · 방향은 원칙 · 폴백은 공식

## AI와 결정론의 경계

| 구간 | 방법 | 이유 |
|---|---|---|
| 거래 의미, 페르소나, 봉투 우선순위, 상품 선택, 코치 발화 | 로컬 EXAONE 판단 | 여러 사실의 맥락과 상충 신호를 함께 읽어야 한다 |
| 세율, 합계, 원화 배분, 소득 통계, 적합성 veto, XP | 결정론 규칙·통계 | 재현성, 법적 정합성, 금액 보존이 우선이다 |
| 흐름과 분기 | Python 오케스트레이션 | LLM이 다음 에이전트나 실행 순서를 임의로 정하지 않는다 |
| 실제 잔액 이동과 외부 심사 연결 | 사용자 승인 | AI는 판정만 하고 금융 실행은 사람이 결정한다 |
| 장애·게이트 실패 | 결정론 폴백 | 로컬 모델이 느리거나 실패해도 핵심 서비스가 중단되지 않는다 |

세금 봉투는 법과 세율표에 따른 결정론 구간이다. 페르소나, 밴딧, LLM이 세율이나 세금 원화를 바꾸지 못한다.

## 데이터에서 서비스까지

```mermaid
flowchart LR
    A["거래·정산·커리어·행동"] --> B["결정론 FactSheet F01–F14"]
    B --> C["UserProfile SSOT"]
    C --> D["긱 소득 구조 2축"]
    C --> E["금융 행동 성향 4축"]
    D --> F["안전자금·예측"]
    E --> G["봉투·페이싱·상품·코치"]
    F --> H["결정론 검증"]
    G --> H
    H --> I["사용자 승인"]
    I --> J["이벤트 로그·다음 제안 보정"]
```

### FactSheet F01–F14

| 구분 | 팩트 | 역할 |
|---|---|---|
| 소득 | F01 월 소득 변동계수, F02 소득원 집중도, F03 입금 간격 중앙값, F04 최장 무수입 공백 | 소득 안정성·소득원 구조와 안전자금 계산의 근거 |
| 소비 | F05 가뭄달 지출 반응, F06 입금 후 몰아쓰기, F07 월 생활비 변동계수, F08 경비 비율, F09 저축 여력률 | 지출 조절·생활비·목표 페이싱의 근거 |
| 행동 | F10 수기 태깅, F11 무수정 승인율, F12 버퍼 조정 방향, F13 커리어 소스 연결, F14 앱 참여 리듬 | 판단 근거와 데이터 품질 측정 |

F14는 네 성향 축의 방향 근거로 사용하지 않고 데이터 충실도만 표시한다. 미션 완료와 커리어 조각 저장도 성향 판독 입력으로 되돌리지 않는다.

### UserProfile SSOT

`app/profile/user_profile.py`가 원장을 한 번 읽어 소득 스트림, 공백, 커리어 신호, 긱 유형, FactSheet, 신선한 페르소나 스냅샷을 합성한다. 배분·페이싱·봉투·상품·코치가 각자 사용자를 재계산하지 않는다.

신규 거래 5건 이상으로 스냅샷이 오래되면 원본은 감사용으로 보존하되 다운스트림 판단에서는 제외하고, 응답에 `persona_used=false`와 신선도 정보를 노출한다.

## 발표에서 보여주는 AI 판단 5지점

| 판단 지점 | 입력 | AI 출력 | 결정론·사람 게이트 |
|---|---|---|---|
| 페르소나 판독 | FactSheet 전체 | 안전 여유·돈의 시간·지출 조절·관리 습관 | 근거 ID, 메뉴 값, 극성·방향 검증, 교정 1회, 중립 폴백 |
| 봉투 추천 | UserProfile, 기존 목표, 또래 통계 | 필요한 목표 봉투 후보와 이유 | 세금·경비는 추천 금지, 중복 제거, 개설은 사용자 |
| 금액 페이싱 | 목표 기한, 사용 가능 금액, 성향, 버퍼 | 보류·기본·당김과 우선순위 | 원화·합계는 순차 배분 엔진, 잔액 이동은 승인 후 |
| 피기 코치 | 검증된 숫자와 라이브 컨텍스트 | 설명·확인 질문·행동 가이드 | 컨텍스트 밖 숫자 차단, 인텐트와 어젠다는 코드가 결정 |
| 상품 매칭 | 적합성 veto를 통과한 하나금융 후보 | 최대 2개 상품과 근거 | 후보 밖 선택 금지, 한도·금리·승인 생성 금지, 실제 심사는 하나원큐 |

거래 분류·해소 질문·이벤트 파싱·강점 선택·인텐트 라우팅은 위 다섯 지점을 보조하는 AI 역할이다.

## 가드레일

### 입력과 근거

- 모델은 원시 데이터 전체 대신 FactSheet와 요약 컨텍스트를 받습니다.
- 판단 결과는 실제 팩트 ID를 인용해야 합니다.
- 인용하지 않은 근거나 존재하지 않는 ID는 폐기합니다.

### 제한된 출력

- 성향 축은 `0.1 / 0.3 / 0.5 / 0.7 / 0.9` 메뉴에서만 선택합니다.
- 강점·상품·스탠스는 서버가 제공한 후보 번호에서만 고릅니다.
- 메뉴 밖 값과 스키마 위반은 교정 재시도 1회 후 폴백합니다.

### 방향 검증

- 각 팩트가 축값을 올리는지, 내리는지, 중립인지 모델보다 먼저 정의합니다.
- 인용 근거의 방향과 최종 값이 반대면 결과를 채택하지 않습니다.
- 안전한 중립 기권은 허용하지만, `low ↔ high` 확신 오판을 차단합니다.

### 숫자와 실행

- 세금·합계·봉투 원화는 결정론 엔진이 계산합니다.
- 코치가 컨텍스트에 없는 3자리 이상 숫자를 생성하면 결정론 문장으로 교체합니다.
- 배분은 `proposed` 상태로 저장되고 `confirm` 또는 `adjust` 이후에만 SQLite 잔액을 바꿉니다.

## 소득과 은퇴 예측

예측 계층은 LLM이 아니라 강건 통계와 시뮬레이션으로 구성됩니다.

- 플랫폼 정산: 채널별 입금 간격의 중앙값
- 반복 발주처: 동일 상대의 재입금 리듬
- 착수금→잔금: 진행 중 계약과 실측 시차
- 신규 일감: 개별 날짜 대신 도착률
- 다음 수입 창: 점 예측이 아니라 P10–P90 구간
- 소득 소멸형 은퇴: 일감 소득이 생활비 아래로 내려가는 시점
- 자금 달성형 은퇴: 누적 자금이 은퇴 필요액에 도달하는 시점

자금 달성형의 4% 인출률과 3% 실질수익률은 데모 가정입니다. 전자는 해외 역사 자료의 표준적 출발점이고 후자는 보수적 prior일 뿐 한국 긱워커에게 검증된 값이 아닙니다. 응답에서 가정을 노출하고 향후 한국 코호트로 보정해야 합니다.

## 학습 루프

학습 배분 정책은 버퍼 원화를 직접 학습하지 않고, 개인이 선호하는 안전 분위수 `P60 / P75 / P90`을 선택합니다.

- prior: 금융 행동 성향의 안전 여유
- 보상: 승인, 거절, 버퍼를 올리거나 내린 조정 방향
- 콜드스타트: 공백 관측이 적을 때 공식과 수축 혼합
- 탐색: 오프라인 백테스트를 통과하기 전까지 비활성화
- 감사: arm, 사후분포, 근거, 사용자 결정 크레딧을 이벤트 로그에 기록

따라서 발표에서는 `데이터가 쌓이면 안전 수준을 보정한다`고 설명하며, 현재 데이터에서 AI가 자동으로 성능을 높였다고 주장하지 않습니다.

## 로컬 서빙과 배포 경계

- 서빙: Ollama
- 모델: EXAONE 3.5 2.4B·7.8B GGUF
- 외부 AI API: 사용하지 않음
- CI: LLM 호출은 mock으로 대체하고 pytest·Ruff·TypeScript typecheck를 실행
- 수동 평가: 로컬 Ollama에서 골든셋 하네스를 실행하고 결과를 append-only 기록
- 확장: 실제 은행 환경에서는 하나금융융합기술원의 금융 특화 파운데이션 모델로 교체 가능한 카트리지 구조

## 발표용 기술 요약

> 거래·정산·커리어·행동 데이터를 F01–F14 FactSheet로 만들고, 단일 UserProfile에서 소득 구조와 금융 행동을 합성합니다. EXAONE은 페르소나·봉투·페이싱·상품·코치를 판단하지만, 세금과 원화는 결정론이 계산하고 실제 실행은 손님이 승인합니다. 모델은 전부 로컬에서 동작하며, 골든셋과 근거·메뉴·방향 게이트로 오판의 전파를 막습니다.
