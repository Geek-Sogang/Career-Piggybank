# Career-Piggybank 백엔드 아키텍처

> 긱워커 가계부 AI 파이프라인의 설계 합의 문서.
> 코드가 진실의 원천이고, 이 문서는 "왜 이렇게 나눴는가"를 기록한다. (2026-07-03 기준)

## 전체 파이프라인

돈이 흐르는 순서(런타임)는 ①→④. 구현은 계약을 먼저 굳히기 위해 역순(③→②→①→④)으로 진행했다.

```
입금/출금 발생
   ↓
① 분류기 (classifier)      "이게 뭔 돈이지?"          라벨 + 확신도 + 근거
   ↓ 라벨된 거래가 쌓임
② 예측기 (spending_profile) "이 사람 평소 패턴은?"      예상 월경비·생활비·소득 변동성
   ↓ 파라미터
③ 배분 엔진 (allocator)     "이번 입금, 얼마씩 나눌까?"  봉투별 금액 제안 (폭포수 룰)
   ↓ 제안
④ 코치 (coach)              "말로 설명 + 확인 요청"      승인/조정/거절 → 봉투 반영
```

## 설계 원칙 (합의 사항)

이 원칙들은 개별 엔진보다 우선한다. 어기는 PR은 머지하지 않는다.

1. **단일 책임** — 분류기는 분류만, 예측기는 숫자만, 배분 엔진은 계산만, 코치는 말만 한다.
   예: "계약금은 보수적으로 다룬다"에서 *계약금 판정*은 분류기, *보수적 배분*은 배분 엔진의
   정책, *그걸 말해주는 것*은 코치의 일이다.
2. **AI는 판단·말, 룰은 돈 계산** — LLM은 산수를 틀리고, 재현이 안 되고, 근거를 못 댄다.
   그래서 금액이 결정되는 모든 구간(세금·배분)은 결정론 룰/통계이고, AI는 텍스트 이해(분류)와
   자연어 설명(코치)에만 쓴다. 사용자 경험은 여전히 "AI의 제안"이다 — 파이프라인 전체가 AI 시스템이고,
   마지막 산수 한 조각만 계산기가 담당하는 것.
3. **제안 → 사람 승인 → 실행** — 엔진의 출력은 '실행'이 아니라 '제안'. 평소 패턴이면 자동 적용
   가능하되, 평소와 다른 입금(평균 1.5배↑)·콜드스타트는 코치가 확인을 요청한다(`needs_confirmation`).
   투자 실행도 동일 원칙(AI는 제안만, 실행은 사람).
4. **설명가능성 — 숨은 계산 금지** — 모든 판정에 근거(`signals`/`reasons`), 모든 계산에 가정
   (`assumptions`)을 응답에 노출한다. 근거 없는 판정은 테스트로 금지한다.
5. **확신 없으면 찍지 않는다** — 분류 확신도가 낮으면 자동 반영하지 않고 `needs_review`로
   사람에게 넘긴다(수기 태그 UI). 조용히 틀리는 것보다 물어보는 게 신뢰를 지킨다.
6. **AI는 로컬/온프렘 오픈소스만** — 은행 망분리 전제. 외부 LLM/AI API 절대 금지. (Ollama 서빙 예정)
7. **세금 계산은 결정론, AI 금지** — 세율·원천징수는 법이다. `tax_envelope.py`는
   모바일 `taxEnvelope.ts`와 미러 — 수정 시 반드시 둘 다 바꾼다.
8. **사용자 피드백이 곧 학습 데이터** — 수기 태그는 분류기의 정답지로, 제안 조정분
   (`adjustment_delta`)은 개인화 재료로 쌓인다. 별도 라벨링 비용 없이 쓸수록 정확해지는 구조
   (기획서 §12 부트스트랩).

## 멀티에이전트 문법 (AI 사용 시 무조건 준수)

LLM을 쓰는 모든 곳은 아래 문법을 따른다. 위반하는 PR은 머지하지 않는다.

- **역할 분리** — 에이전트마다 하나의 일. 분류 에이전트는 분류만, 판정(judge) 에이전트는
  검증만 한다. 하나의 프롬프트에 여러 역할을 몰지 않는다.
- **오케스트레이션** — 흐름은 LLM이 아니라 코드가 정한다. 분류 결과(조건 분기)로 다음
  에이전트를 고르고, 룰이 잡은 거래는 LLM을 호출조차 하지 않는다(캐스케이드).
- **가드레일** — 위험한 판단은 AI가 판정만, 실행은 사람.
  · LLM 경로 확신도는 상한(`LLM_CONFIDENCE_CAP=0.75`) — 결정론 룰(0.9~0.95)보다 항상 낮다.
  · 판정 에이전트가 반려하면 자동 반영 금지(`needs_review`).
  · **LLM의 매출(income) 판정은 자동 반영 금지** — 세금·배분을 트리거하는 위험 판단.
    (골든셋 평가에서 개인 입금 오분류를 잡아내 추가된 가드레일)
  · **고액 거래(100만원↑)는 AI가 인지만 하고, 판정 결과와 무관하게 사용자에게 직접 물어본다.**
  · LLM 다운/스키마 위반 시 룰 결과로 안전 폴백 — LLM이 죽어도 서비스는 산다.

## 품질 관리 (QC)

- **골든셋 평가** — `evals/golden_set.json`(라벨 35건+) + `python -m evals.run_eval [--llm]`.
  핵심 지표: **자동결정 정확도 100% 강제**(자동 반영이 틀리는 건 조용한 오류 = 신뢰 파괴,
  애매하면 review가 정답) + 커버리지(자동결정 비율, 수기 태그 부담의 역수).
  현재: 룰만 100%/77%, 룰+LLM 100%/83%. 라벨 변경·삭제는 PR 리뷰 필수.
- **CI 게이트** — 모든 PR에서 pytest(골든셋 룰 게이트 포함)+ruff+모바일 tsc 자동 실행.
  LLM 테스트는 전부 목(mock) — CI에 Ollama 불필요. `--llm` 평가는 로컬에서 수동.
- **무수정 승인율** — `GET /v1/allocations/metrics`. 배분 제안을 사용자가 안 고치고
  승인한 비율 = 시스템 전체 성능의 한 숫자 KPI.

## ① 분류기 — `app/services/classifier.py`

**역할**: 거래 1건 → `kind`(income/expense/living/unknown) + `subtype` + 확신도 + 근거.
**하지 않는 것**: 배분 판단, 금액 계산, 사용자 설명.

긱워커는 발주처가 매번 바뀌므로 일반 가계부의 '상호명 사전' 방식이 안 통한다.
대신 **구조적 신호의 캐스케이드**를 쓴다 (위에서 잡히면 아래로 안 내려감):

| 층 | 신호 | 판정 | 확신도 |
|---|---|---|---|
| 입금 1 | **3.3% 역산** — 입금액÷0.967이 만원 단위로 딱 떨어짐 (원천징수 흔적) | income / settlement(잔금) | 0.95 |
| 입금 2 | 정산 플랫폼 입금자 (크몽·위시켓…) | income / settlement | 0.90 |
| 입금 3 | 사업자 형태 이름(㈜·스튜디오…) + 원천징수 흔적 없음 | income / **advance(계약금 의심)** | 0.70 |
| 입금 4 | 개인 이름 형태 (한글 2~4자) | **unknown → 수기 태그** | 0.40 |
| 입금 5 | 무신호 (페이 정산 등) | **unknown → 수기 태그** | 0.30 |
| 출금 1 | 구독/SaaS 키워드 (Figma·AWS…) | expense / subscription | 0.90 |
| 출금 2 | 운영비 키워드 (호스팅·장비…) | expense / operating | 0.75 |
| 출금 3 | 기본값 | living (생활 지출) | 0.70 |

- 3.3% 역산은 이 서비스의 검증축(3.3% 신고소득)과 같은 데이터 — **분류기가 곧 검증 엔진**이라는 스토리.
- `REVIEW_THRESHOLD(0.65)` 미만이면 `needs_review=True` — 프론트 가계부의
  "토스페이 정산 · AI 미분류" 행이 정확히 이 케이스다.
- **2단계(구현됨)**: `classifier_llm.py` — unknown만 로컬 LLM 폴백. 분류 에이전트(few-shot)
  → 판정 에이전트 체인, 위 멀티에이전트 문법의 가드레일 전부 적용. 룰 계층은 그대로 —
  LLM이 죽어도 데모가 안 죽는다.

API: `POST /v1/classify` (배치, `review_count` 포함, `llm_fallback: true`로 LLM 경로 활성화)

## ② 예측기 — `app/services/spending_profile.py`

**역할**: 라벨된 거래 이력 → 배분 엔진의 입력 파라미터.
**하지 않는 것**: 라벨 판단, 배분. **AI 아님** — 전부 결정론 통계 (파이썬 `statistics`뿐).

- **중앙값 기반** — 긱워커 소득·지출은 이상치가 많아 평균이 속는다.
  (대박 달 경비 250만이 껴도 예상 경비는 33.5만으로 유지)
- **소득 변동계수(CV = 표준편차/평균)** — 배분 엔진의 버퍼 목표 개월 수 재료.
- **콜드스타트**: 이력 3개월 미만이면 직군 프리셋(developer/designer/creator)과 가중 혼합
  (`blend_weight` = 개월수/3), 쌓일수록 개인 통계로 자연 전환. → "같은 입금인데 가입 첫날과
  3개월 후 제안이 달라진다"가 개인화의 데모 증명 장면.

API: `POST /v1/profile/estimate` — 응답 `profile`을 그대로 `/v1/allocations/propose`에 연결.

## ③ 배분 엔진 — `app/services/allocator.py`

**역할**: 입금 1건 + 프로필 + 봉투 잔액 → 봉투별 금액 **제안**.
**하지 않는 것**: 실행(승인 전 봉투 반영 금지), 라벨 판단, 자연어 설명.

**폭포수 룰** — 우선순위대로 갭을 채우고, 남는 전부가 여윳돈:

```
1. 세금봉투   = 입금 × 실효 추가세율        (tax_envelope 결정론, 무조건 먼저)
2. 경비봉투   = 예상 월경비까지 부족분만
3. 즉시가용   = 예상 월생활비까지 부족분만
4. 여윳돈     = 남는 전부 (+ 반올림 오차 흡수 → 합계 == 입금액 불변식)
```

- "많이 들어온 달은 여윳돈 비중이 커진다"는 별도 로직이 아니라 폭포수의 성질로 나온다.
- 버퍼 목표 = 생활비 × (1 + 4×CV) 개월, 1~6개월 클램프. 목표 초과분 = `invest_available`
  (여윳돈 투자 제안 가능액 — 즉시가용·세금봉투는 절대 건드리지 않음).
- **제안/승인 상태기계**: `proposed → confirmed / adjusted / rejected`.
  조정 시 합계=입금액 검증, 조정분 `adjustment_delta` 기록(개인화 피드백).

API: `POST /v1/allocations/propose` → `POST /v1/allocations/{id}/decision` → `GET /v1/allocations/{id}`

## 세금 엔진 — `app/services/tax_envelope.py` (기존)

종소세 누진세율표 + 3.3% 원천징수 결정론 계산. **AI 금지 구간.**
모바일 `apps/mobile/src/lib/taxEnvelope.ts`와 미러 — 수정 시 반드시 양쪽 동기화
(온디바이스 오프라인 데모를 위해 의도된 중복).

API: `POST /v1/tax-envelope/annual`, `POST /v1/tax-envelope/split`

## ④ 코치 — `app/services/coach.py`

**역할**: 배분 근거·분류 신호(이미 한국어 문장으로 나옴)를 대화체로 전달, 확인 요청, 질문 응답.
**하지 않는 것**: **숫자 계산 절대 금지** — 모든 금액은 결정론 엔진이 계산해 완성된 값으로
프롬프트([컨텍스트])에 주입하고, LLM은 말만 다듬는다.

- **결정론 숫자 검증기**: 응답의 3자리 이상 숫자가 컨텍스트에 없으면(지어냈으면) 응답을
  버리고 근거(reasons) 기반 템플릿으로 교체 — 할루시네이션 금액이 사용자에게 닿는 경로 차단.
- LLM 다운 시에도 근거 템플릿 폴백 — 코치가 죽어도 서비스는 산다.
- 응답에 `source`(llm/fallback)·`verified` 노출.

- 서빙: Ollama(맥 로컬, 망분리 전제 충족) + **EXAONE 3.5 7.8B Q4_K_M**(LG 한국어 특화
  오픈 모델, `app/core/config.py`의 `ollama_model`) — 분류기 LLM 폴백과 공유.
- 컨텍스트는 원시 거래내역이 아니라 요약된 프로필만 (짧을수록 로컬 모델이 정확·빠름).

## 데모용 상수 (실데이터 보정 필요)

구조는 검증된 원칙(봉투 예산법·폭포수 배분·비상금 n개월 룰·강건 통계)이지만,
아래 계수는 데모용 가정치다. 전부 `assumptions`로 응답에 노출되며, 파일럿 데이터로 보정한다.

| 상수 | 값 | 위치 |
|---|---|---|
| 버퍼 개월 = 1 + 4×CV (1~6 클램프) | `BUFFER_CV_SLOPE=4` | allocator |
| 코치 확인 임계 = 평균 입금 × 1.5 | `WINDFALL_RATIO=1.5` | allocator |
| 콜드스타트 졸업 | `COLD_START_MONTHS=3` | spending_profile |
| 직군 프리셋 수치 | `PERSONA_PRESETS` | spending_profile |
| 수기 태그 임계 | `REVIEW_THRESHOLD=0.65` | classifier |
| 역산 단위/오차 | 만원 / ±5원 | classifier |
| LLM 확신도 상한 | `LLM_CONFIDENCE_CAP=0.75` | classifier_llm |
| 고액 직접확인 임계 | `LARGE_TXN_REVIEW=100만원` | classifier_llm |
